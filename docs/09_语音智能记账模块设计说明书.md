## 1. 模块概述

### 1.1 业务背景

校园二手交易平台每天都有大量学生发布教材、数码、服饰、生活用品等商品。**90% 的发布动作发生在 5 分钟内**——用户用手机随手拍一张图、写几个字，就希望平台帮他完成剩下 80% 的「琐事」：

- **识别类目**：这是教材、数码还是服饰？
- **起个标题**：20 字以内，能在信息流里脱颖而出。
- **写描述**：60 字以内，说清成色、瑕疵、是否包邮。
- **估个价格**：参考同款历史成交，别定得太离谱。
- **做内容审核**：违禁词 / 色情 / 广告 / 黑产导流要前置拦截。
- **促成议价**：买家聊「在吗？」「能便宜点吗？」时，卖家需要一个「拟人化」的建议回复。
- **通用问答**：用户进入「易物助手」时，可以问「怎么上架？」。

旧版「语音智能记账」（FF-VOICE-001）模块以家庭收支为核心场景，强调 ASR → 解析 → 记账。本版本完全废弃，**整体业务切换为「图片 + 文字混合输入 → AI 助手多任务输出」**，承担智能发布 / 价格 / 审核 / 润色 / 议价 / 客服 / 问答 7 类任务。

### 1.2 模块定位

| 维度 | 说明 |
|------|------|
| **产品定位** | 平台差异化竞争力，**降低发布门槛 + 提升交易效率**。 |
| **技术定位** | 业务模块之一，对外通过 `/api/ai/*` 暴露能力；不直接耦合 Django ORM，可被独立部署与单测。 |
| **依赖定位** | 仅依赖 `market.services.llm_client`（HTTP 调用）、`market.services.ai_prompts`（Prompt 模板）、`market.services.ai_data_context`（构造上下文），与具体业务 model 解耦。 |
| **降级定位** | 任意环节失败（无 Key / 限流 / 网络 / JSON 解析失败）都必须降级到本地规则，并标注 `is_ai_fallback=true`，**不阻塞主流程**。 |

### 1.3 设计目标

1. **可用性**：未配置 LLM Key 时所有端点 100% 可用（返回降级结果）。
2. **可解释**：每个返回都带 `confidence` / `reasoning` / `sample_count` / `is_ai_fallback`，前端可向用户解释。
3. **可审计**：调用记录写入 `ai_call_logs`（未来扩展），便于追踪成本与质量。
4. **可扩展**：新增 AI 任务只需在 `ai_prompts.py` 加模板 + 在 `ai_service.py` 加方法，**不改动 views 层**。
5. **可降级**：失败时绝不抛异常到前端，错误信息统一为「AI 暂不可用：原因」。

### 1.4 不在本模块范围内

| 范围外项 | 描述 |
|----------|------|
| 用户私聊消息的智能回复 | 属于「会话模块」（`conversations`）的 LLM 增强；本模块只提供 `customer-service` 给卖家用。 |
| 图片的 CDN 转码 / 缩略图 | 属于上传模块（`upload_views.py`）。 |
| 推荐商品（协同过滤） | 属于 `recommend` 服务（v2.0 规划）。 |
| 语音 ASR | 旧版能力；v1.0 起所有 AI 入口仅接受 **图片 URL / 图片 base64 / 文本**，**不再支持音频**。 |

## 2. 设计原则与目标

### 2.1 设计原则

| 原则 | 说明 |
|------|------|
| **人机协同** | AI 产出永远不是「最终结果」，前端必须展示给用户确认。 |
| **永不抛错** | AI 模块失败 = 降级 + 标 fallback + 走本地规则。 |
| **数据驱动** | 价格建议 / 议价 / 关键词提取都基于真实成交数据 + 真实商品字段。 |
| **可解释** | 必须返回 `reasoning` 字段，让用户知道 AI 凭什么给出这个价。 |
| **成本可控** | 关键字 / 关键词先用本地规则；LLM 只在「理解类」任务调用。 |
| **结果确定** | 同一份输入 + 同一份 prompt + 同一份 LLM 必须返回**可重现的 JSON**（temperature ≤ 0.5）。 |

### 2.2 与非功能需求的对齐

| 需求类别 | AI 模块响应 |
|----------|------------|
| 性能 | 单次 LLM 调用 95p < 2.5s；含降级 < 200ms。 |
| 限流 | 60 req/min/user；管理员可调整。 |
| 隐私 | 不向 LLM 发送真实用户姓名、学号、手机号、地址；仅发送脱敏的 ID。 |
| 合规 | 内容审核对色情 / 暴力 / 政治 / 导流四类高危词必须 0 漏检。 |
| 可用性 | 无 Key 时降级成功率 100%；LLM 5xx 时降级成功率 100%。 |

### 2.3 边界与不变量

1. `AiService` 的所有公开方法**只返回 dict**（与 dataclass 的 `to_dict()` 一致），便于视图层直接 `ok(data)`。
2. 任何抛出异常都会被 `try/except` 捕获并降级，**不会**让 5xx 暴露给前端。
3. LLM 返回的 JSON 字符串被 `parse_json_from_content` 容忍 markdown 包裹；解析失败同样降级。
4. `confidence ∈ [0, 1]`；越界自动截断。
5. 任何价格字段 `float → round(., 2)`，避免浮点尾差。

### 3.2 调用流程（以 `publish-assist` 为例）

```
[小程序]   1. 用户选择图片 / 输入文字 / 调 wx.uploadFile 上传 → 拿到 image_url
           2. wx.request POST /api/ai/publish-assist/  { image_url, draft_text }
[views]    3. AiPublishAssistView.post 校验参数（至少 1 项）
           4. ai = get_ai_service() 单例
           5. ai.publish_assist(image_url=..., draft_text=...)
[service]  6. 调 build_publish_context(draft_text)  拼装历史价 / 类目 / 描述
           7. 选 PUBLISH_ASSIST_PROMPT，.format(image_note, draft_text, price_history)
           8. 构造 messages：[system, user(multimodal)]
           9. self.llm.chat(messages, temperature=0.2, max_tokens=600)
           10. parse_json_from_content(content) → dict
           11. _normalize_publish_result(data)  校验 category/condition/price
           12. PublishAssistResult.to_dict()  返回
[views]    13. ok(result, 'AI 识别完成') → 200 {code:0, message, data}
[小程序]   14. 前端展示「AI 智能识别成功」徽标；用户可编辑后再点「正式发布」
```

### 3.3 失败兜底

```
第 9 步 llm.chat 抛 LlmException(50301 未配置 / 50302 返回空 / 5xx)
↓
catch 块 → _mock_publish_assist(draft_text)  本地规则类目识别
↓
return PublishAssistResult(... is_ai_fallback=True, confidence=0.35)
↓
前端响应：is_ai_fallback=true  灰色「AI 推荐」标识，告知「当前为本地智能推荐」
```

## 5. AI 一键发布（publish-assist）

### 5.1 业务目标

让用户在 3 步内完成商品发布：

1. 拍 1 张图（可选 base64 内联）。
2. 写 1 句简短描述。
3. 点击「AI 一键识别」，前端自动填好：类目、标题、描述、价格、成色、标签。

> 用户仍可在 AI 识别后修改任意字段，最后点「正式发布」。

### 5.2 请求

```http
POST /api/ai/publish-assist/ HTTP/1.1
Authorization: Bearer <access_token>
Content-Type: application/json
```

```json
{
  "image_url":  "https://cdn.xxx.com/products/p1_1.jpg",
  "draft_text": "九成新 iPad，几乎没怎么用过",
  "image_b64":  null,
  "image_mime": "image/jpeg"
}
```

字段说明：

| 字段 | 必填 | 描述 |
|------|------|------|
| `image_url` | 三选一 | 已上传图片的公网 URL（推荐，节省 LLM token）。 |
| `image_b64` | 三选一 | 当 LLM 服务在内网无法访问 CDN 时直接传 base64。 |
| `draft_text` | 三选一 | 用户手写文字，可与图片同时给。 |
| `image_mime` | 否 | `image/jpeg`（默认）/ `image/png` / `image/webp`。 |

> 校验：至少提供一个字段；超过 200 字符的 `draft_text` 自动截断。

| 字段 | 类型 | 约束 | 备注 |
|------|------|------|------|

### 5.6 Prompt 模板

完整内容见 [附录 A.1](#附录-a1publish_assist_prompt)。

### 5.7 降级策略

`AiService._mock_publish_assist(draft_text)`：

1. 从 `draft_text` 中匹配中文类目关键词（`书/教材/考研/英语/高数` → 教材；`手机/iPhone/华为/小米/iPad/电脑/笔记本/耳机` → 数码；`上衣/裤子/裙子/外套/鞋/卫衣/T恤/衣服` → 服饰；`水杯/台灯/收纳/自行车/椅子/桌子/伞/生活` → 生活；其他 → `其他`）。
2. 从 `draft_text` 中匹配成色关键词（`全新/未拆` → 全新 factor=1.0；`九成/9成` → 9 成新 factor=0.85；`八成/8成` → 8 成新 factor=0.7；默认 `9成新` factor=0.85）。
3. 计算 `price = base * factor`，`base` 在类目中给定（教材 25、数码 800、服饰 50、生活 35、其他 30）。
4. 标题截断 `draft_text` 前 20 字；描述截断前 60 字。
5. `confidence = 0.35`（标定为「弱推荐」）。

- 输入图片时禁用「发布」按钮，loading 5 秒。

## 7. AI 内容审核（moderate）

### 7.1 业务目标

发布商品、私聊、评论时，前端调用一次 `/ai/moderate/`，对文本做风险评估。**任何高风险词必须实时拦截**（不允许进入数据库）。

### 7.2 请求

```http
POST /api/ai/moderate/ HTTP/1.1
Authorization: Bearer <access_token>
Content-Type: application/json
```

```json
{ "text": "诚心出 iPad，加我微信 abc123 详聊" }
```

### 7.3 响应

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "safe":         false,
    "risk_level":   "medium",
    "reasons":      ["命中违规词：加微信"],
    "suggestion":   "人工审核",
    "is_ai_fallback": true,
    "message":      "本地关键词预警"
  }
}
```

### 7.4 风险等级

| 等级 | 描述 | 前端动作 | 后端动作 |
|------|------|----------|----------|
| `none` | 无风险 | 放行 | 写入 |
| `low` | 擦边（如「加微信」「兼职」） | 黄色「请遵守平台规则」提示 | 写入 + 标记 |
| `medium` | 软违规（色情暗示、站外导流） | 红色「需修改后发布」弹窗 | 拦截 + 记录 |
| `high` | 硬违规（毒品、枪支、代考） | 红色「禁止发布」并上报 | **强制拦截** |

### 7.5 二级审核管线

```
[text 输入]
   │
   ▼
[本地关键词]   ← _quick_keyword_check  毫秒级
   │
   ├─ 命中 high_risk → 直接返回 safe=false, risk_level=high
   ├─ 命中 medium_risk → 直接返回 safe=false, risk_level=medium
   ├─ 命中 low_risk → 直接返回 safe=true, risk_level=low
   └─ 未命中 ↓
   │
[LLM 精细审核]  ← 仅在本地未命中时调用，节省配额
   │
   └─ 失败/未配置 → safe=true, risk_level=low, is_ai_fallback=true
```

### 7.6 关键词词库维护

| 类别 | 词例 |
|------|------|
| `high_risk` | 毒品、冰毒、海洛因、大麻、枪支、弹药、管制刀具、代考、替考、代写论文、办证、办假证 |
| `medium_risk` | 微商、兼职日结、约炮、一夜情、陪睡、裸聊、高仿、A 货、盗版、破解版、枪版 |
| `low_risk` | 加微信、加我、私聊、二维码、兼职 |

> 词库升级走 PR：词条 + 风险等级 + 命中示例。每月评审一次。

### 7.7 Prompt 模板

完整内容见 [附录 A.3](#附录-a3content_moderate_prompt)。要点：

- 要求 LLM 严格输出 JSON：`{safe, risk_level, reasons, suggestion}`。
- 风险等级必须为 `none / low / medium / high` 之一。
- `reasons` 数组中每条 ≤ 20 字，最多 10 条。

### 7.8 降级策略

- 未配置 LLM：返回 `safe=true, risk_level=low, is_ai_fallback=true`，但**不写入** `audit_logs` 风险项。
- 真实 LLM 异常：同上 + 在 message 字段附异常摘要。
- 本地关键词命中 high/medium：直接拦截，**不再调 LLM**。

### 7.9 前端集成

- 发布商品提交前：先 `await api.aiModerate({ text: title+'\n'+description })`。
- `safe=false` 时弹窗「内容需修改」+ 列出 `reasons[]`，按钮置灰。
- 私聊发送前：同流程（仅当消息含 `@all` 关键词或长度 > 50 触发，正常闲聊不审核以免打扰）。

## 10. AI 关键词提取（extract-keywords）

### 10.1 业务目标

从商品标题 + 描述中提取 3-6 个搜索关键词与 2-4 个短标签，供：

- **搜索**：补充分词后的关键词。
- **推荐**：做 item-based 相似度。
- **去重**：发布前检查是否与已有商品重复。

### 10.2 请求

```http
POST /api/ai/extract-keywords/ HTTP/1.1
Authorization: Bearer <access_token>
Content-Type: application/json
```

```json
{
  "title":        "iPad Air 5 自用，几乎全新",
  "description":  "电池循环 30 次，配件齐全，欢迎自取验货",
  "category":     "数码",
  "max_keywords": 6
}
```

### 10.3 响应

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "keywords":       ["iPad", "Air5", "电池循环", "自用", "配件齐全"],
    "tags":           ["iPad", "Air5", "自用", "全新"],
    "is_product":     true,
    "category_guess": "数码",
    "is_ai_fallback": false,
    "message":        "AI 关键词提取完成"
  }
}
```

### 10.4 关键词规整化

LLM 输出后在服务层做规整化：

1. 去空字符串、strip。
2. 去重（保持顺序）。
3. 截断：`keyword` ≤ 20 字符；`tag` ≤ 6 字符。
4. 上限：`max_keywords=6`（默认 6，最大 12）；`tags ≤ 4`。
5. `category_guess` 必须在 `VALID_CATEGORIES = {教材, 数码, 服饰, 生活, 其他}` 内；否则 fallback 到请求中的 `category` 或 `其他`。

### 10.5 本地兜底（`_local_extract_keywords`）

不依赖 jieba 的简易算法：

1. 用正则 `[\u4e00-\u9fa5]{2,6}|[A-Za-z0-9\-]{2,20}` 抓 token。
2. 过滤 `_STOPWORDS`（停用词表，~30 个常用中文虚词）。
3. 标题里的 token 权重 +1。
4. 按频次倒序、长度短的优先；取前 8 个做 `keywords`，前 4 个做 `tags`。

### 10.6 降级策略

- 输入全空 → 返回空 `keywords / tags`，`is_product=false`。
- LLM 未配置 → 用本地兜底，`is_ai_fallback=true`。
- LLM 异常 → 用本地兜底 + message 附异常。

### 10.7 前端集成

- 不直接面向用户。
- 服务端在「保存草稿」与「正式发布」两个时机自动调用，结果存入 `Product.tags[]` 字段（按需求保留）。

### 13.1 LlmClient 公共能力

### 13.2 LlmException 错误码

| 错误码 | 含义 | 触发场景 |
|--------|------|----------|
| 50301 | LLM 未配置 | `is_configured=false` |
| 50302 | LLM 返回空 | content 为空字符串 |
| 50303 | 网络超时 | `requests.exceptions.Timeout` |
| 50304 | 限流被拒 | HTTP 429 |
| 50305 | 鉴权失败 | HTTP 401/403 |
| 50306 | 服务端错误 | HTTP 5xx |
| 50307 | JSON 解析失败 | `parse_json_from_content` 抛错 |
| 50308 | 内容审核违规 | 文本过长 / 包含敏感词 |

1. 所有 prompt 模板**只放 `ai_prompts.py`**，**禁止散落在 views**。

## 14. 数据上下文构建

`market.services.ai_data_context` 负责「为 LLM 拼装真实业务数据」，避免 prompt 漂在「常识」里。

### 14.1 `build_publish_context(draft_text)`

| 步骤 | 描述 |
|------|------|
| 1 | 关键词抽取：从 `draft_text` 抓 2-4 字 token。 |
| 2 | 类目猜测：与 5 个一级类目的种子词做重合度匹配。 |
| 3 | 调 `get_price_history(category, days=30)` 拿历史价摘要。 |
| 4 | 拼装 `summary_text`：「近 30 天 N 笔成交：¥min - ¥median - ¥max（成色：xxx）」。 |

### 14.2 `build_price_suggest_context(category, condition, days)`

| 步骤 | 描述 |
|------|------|
| 1 | 查 `Order` 表（`status in (confirmed, completed)` 且 `created_at >= now()-days` 且 `product__category=category` 且 `product__condition=condition`）。 |
| 2 | 聚合：`sample_count, price_min, price_median, price_max, price_avg`。 |
| 3 | 返回 dict 给 prompt 渲染。 |

### 14.3 `format_history_for_prompt(history, max_turns=12)`

| 步骤 | 描述 |
|------|------|
| 1 | 截取最近 `max_turns` 轮。 |
| 2 | 每条 role 标准化为 `buyer / seller / system`。 |
| 3 | 渲染为 `买家: xxx\n卖家: yyy\n...` 文本。 |
| 4 | 文本超过 1500 字时按 turn 等比截断。 |

### 14.4 `get_price_history(category, days)`

封装 14.2 的查询逻辑，供 `negotiate_price` / `price_suggest` 共用。

## 15. 降级策略与容错

### 15.1 失败分类

| 失败类型 | 是否降级 | 降级方式 |
|----------|----------|----------|
| LLM 未配置（Key 缺失） | ✅ | 走本地 mock 规则 |
| 网络超时（5s+） | ✅ | 走本地 mock 规则 |
| 限流（429） | ✅ | 走本地 mock 规则 + 警告日志 |
| 鉴权失败（401/403） | ✅ | 走本地 mock 规则 + **错误告警**（管理员需修复） |
| 服务端错误（5xx） | ✅ | 走本地 mock 规则 + 错误告警 |
| JSON 解析失败 | ✅ | 走本地 mock 规则 |
| 业务校验失败（如 category 不在枚举） | 部分 | 自动归一化（不在枚举 → `其他`） |
| 限流（60/min/user） | ❌ | 抛 `Throttled`（42900） |

### 15.2 降级时序

```python
try:
    if not self.llm.is_configured:
        raise LlmException('LLM 未配置', code=50301)
    content, _ = self.llm.chat(messages, temperature=0.2, max_tokens=600)
    data = self.llm.parse_json_from_content(content)
    result = self._normalize_publish_result(data)
    result.is_ai_fallback = False
    return result.to_dict()
except Exception as exc:
    logger.warning('publish_assist 降级: %s', exc)
    fallback = self._mock_publish_assist(draft_text=draft_text)
    return fallback.to_dict()
```

### 15.3 告警与监控

- 关键告警：`logger.error` 写 `ai_service` logger，文件 + Sentry 双通道。
- 关键指标（Prometheus）：
  - `ai_call_total{task, status}`（status=ok/fallback/error）。
  - `ai_call_latency_seconds{task}`（P50/P95）。
  - `ai_token_usage_total{task, model}`。
- 降级比例 > 30% 时自动飞书告警给 SRE。

## 16. 安全、限流与审计

### 16.1 限流策略

| 端点 | 限流 | 说明 |
|------|------|------|
| `/api/ai/*`（除 health） | 60/min/user | 防单用户刷 token |
| `/api/ai/health/` | 30/min/ip | 探活用 |
| `/api/ai/moderate/` | 120/min/user | 聊天/发布高频，单独放宽 |
| `/api/ai/chat/` | 60/min/user | 易被刷 |

### 16.2 隐私保护

- 不向 LLM 发送：
  - 用户真实姓名、昵称、学号、手机号、邮箱。
  - 商品图片中的人脸（用 `mask_user_face` 服务预处理，但 v1.0 仅在 `customer-service` 调用时做）。
  - 私聊中的身份证号、银行卡号、地址。
- 发送内容必须经 `_sanitize(text)`：
  - 手机号正则 `1[3-9]\d{9}` → `***`。
  - 微信号正则 `[a-zA-Z][a-zA-Z\d_-]{5,19}` → `***`。
  - 邮箱正则 → `***`。

### 16.3 审计

- 每次 AI 调用写 `ai_call_logs`（v1.1 上线）：
  - `user_id, task, latency_ms, is_ai_fallback, model, prompt_tokens, completion_tokens, error_code`。
- 管理后台 `/admin/ai/logs` 分页查询（v1.1 规划）。

### 16.4 内容合规

- 内容审核在 LLM 调用前**先过本地关键词**（毫秒级）。
- 高风险词命中即直接拦截，**不调 LLM**（节省 token + 防止绕过）。
- 管理后台「举报」页对举报内容二次审核。

### 18.3 月度成本估算

- 假设 DAU 2000，每人每天调 AI 平均 4 次 = 8000 次/天。
- 平均单次成本 ¥0.012（gpt-4o-mini 量级）。

- 99% 请求 < 3.5s。

```text
【用户文字描述】: {draft_text}
【同类历史价（参考）】: {price_history}

1. 必须严格输出 JSON，不要加任何 markdown 标记、不要 ```json``` 包裹
2. 描述使用第二人称视角（"您可..."、"日常通勤..."）
3. 标题要吸引人但要真实，不夸大、不诱导
4. 价格根据物品新旧程度和同类商品市场合理定价；
   若提供了历史价参考，请落在其 ±15% 区间内（无参考则按常识）
5. 如果图片不清楚或不是商品（自拍/风景/食物/截图等），confidence 必须 < 0.5，
   category 设为 "其他"，title 提示"非商品图片"
6. tags 必须是 3-5 个中文短词，不要带 # 符号
7. condition 只能取 4 个枚举值之一
```
